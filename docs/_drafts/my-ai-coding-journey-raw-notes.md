# 我用 AI 寫程式的四年 — 原始筆記

> 這份文件保存討論過程中收集的原始素材，供日後參考。

---

## 用戶原始草稿

我是2023年開始用ChatGPT也是那時候開始寫React，我大多都是請他寫範例程式碼，理解之後再改寫成我的風格。

2024年開始用Github Copilot，幫我開發一些擴充性的需求、寫測試、寫commit

2025年換了工作之後改用Claude Code，我把大量的頁面需求交給他開發，原本自己要做一兩個月的事情，縮短了非常多時間。

2026年，我幾乎沒什麼自己手寫程式碼了，

---

## 各年份補充細節

### 2023

我當時從露天轉調到 PChome，一上工就要負責新後台的架構，當時我還剛學習 React 一個月左右, 但是透過 ChatGPT 教我寫 React，還有主管的 code review，我在一個多月就把新後台的第一個系統上線了

我仍然需要 google 一些 AI 都解不了的 bug，而且整個前端架構還是需要自己實驗合適的套件、選出最適合我們的方案，但這個時期寫程式已經不再是自己摸索，自己看很多別人寫的程式碼了。

### 2024

2024 年 當時認為每次都在 chatGPT 討論程式碼很麻煩，尤其是 context 越大，他給的程式碼會越來越歪，同時 cursor 跟 windsurf 這兩個從 vscode 分支的 ai ide 變得很流行，我實際使用後還是不太習慣，所以投入了 github copilot 的懷抱

ai 直接在編輯器裡改檔案真的解決了很多開發上的需求，當時也特地買課程學習怎麼使用 github copilot, 我發現不只是幫我改檔案，他還可以幫我寫 git commit, 如果 spec 寫得夠清楚的話，直接幫我把一個功能做完也是辦得到的。

我跟團隊還幫後台加上單元測試、E2E 測試，當時還可以在 PR 呼叫 copilot 幫忙 code review

我那時候就發現，用 AI 開發真的減輕很多認知負擔，我連績效考核都用 AI 幫我修飾排版跟用詞，雖然可能花更多時間去審核 ai 的產出，但我對結果大多都是滿意的。

### 2025

2025 年上半年離職休息了一陣子，後來再求職的時，只要有作業我就會用 AI 協助我開發，作業規定一個禮拜交，我兩天就完成大部分功能，剩下我自己處理的事情只有調整樣式、整理程式碼。

後來加入一間新公司，一開始也是負責一個新後台系統，接手這個系統的時候發現程式碼架構大多都不是 react 或 js 的 best practice, 後來得知大多程式碼都是 Claude 寫的，但是看起來就很難以維護

Claude 很喜歡寫物件導向的寫法，很多時候方法都包在 class 或 物件裡，這樣的寫法不是不行，但一個頁面同時載入太多用不到的方法，只是佔用記憶體而已，而且這跟我熟知的 react hook, js 的開發思路差太多了。

我花了快一個月才把整個後台重構成現代的前端架構，同時我一邊學習 Nextjs 這個框架，這時還只有用 ChatGPT 幫我

後來聽聞很多朋友說 Claude Code 很好用，於是我也想說來試用一個月看看，才第一週我就回不去了，以前在用 Copilot 的感覺是，開發過程中我還是要很關注我的程式碼，我決定我的程式碼會長怎樣。但我用 Claude Code 之後，他已經可以幫我把大部分程式碼個雛形寫好，我大多時候是跟他在架構、程式碼風格來回討論，因為太興奮了還寫了一篇部落格。

後來我的工作就慢慢的不只是 coding 了，我還研究了怎麼下 prompt 研究 slash command 再到後來 sub agents 跟 skills，如何寫 claude.md 如何讓 ai agent 更遵從指示等等

因為 claude code pro 的用量很容易用完，所以我也接著使用 codex 跟 gemini，一開使這兩個產品還不如 claude code 那麼順手，但是 2025 下半年卻爆炸性的趕上了。

我也總結出各家模型的特性，claude 很會寫計劃也很會做大部分日常的開發。 gpt-5 codex 寫程式很厲害，但叫他做文件、做大一點的功能就會偷懶。gemini 特別會寫文件，用得順的時候我覺得跟 claude 不相上下，但常常腦補或是秀斗，容易做錯方向或是做到一半卡住。

光是 2025 我就有太多 ai 的話題可以聊了，有時候我還會覺得我是前端工程師嗎？我好像在做下一個層級的工作了。

### 2026

隨著各家的模型的理解力提升到一個層次，我發現只要訂好規則他就會有很高的遵從性，並且我也養成很好的 prompt 習慣

每次只要開發我都會先整理好我的需求及想法，再請 ai agent 理解後寫計劃的 md 檔，當我通過計劃請他開始做，我只要等最後的結果就好了，我只會簡單 review 過 AI 的成果，我會注意他每次是否有照開發規範處理，或是如果他走偏了我會判斷他是缺乏更多資訊，或是指示不夠明確就重新再討論一次。我大多時候都是跟 ai 討論需求跟目標，而不是看程式碼，寫程式這件事情已經被壓縮到要用很少的時間處理了。

## 關鍵感受

- 2026 年的感受：欣然接受，這是進化
- 新角色：明確需求 / 拆解問題、楚局觀 / 架構設計、調教 AI 的方向 / 反覆溝通
- 文章目的：紀錄，就是想把這小段歷程寫下來

## 結論討論（原始想法）

我覺得今年開始有一種感覺，我負責業務邏輯的部分，而 AI 則是負責程式碼邏輯的部分，雖然我還是處理很多前端的工作，但我現在掌握的不是程式碼寫得多好多有效率，而是有沒有達到前端應有的架構跟品質。

而且不只是前端開發，我也有遇過後端 Nginx 的問題，跟後端工程師溝通這個問題的過程，也是先用 AI 了解整個可能的因素，之後做成 task 寫回 Jira 的票上，快速的完成整個問題回報。

在開發功能的過程中如果也有更好的做法，我也會先請 AI 幫我做成計畫，之後再執行。

因為寫程式這件事被 AI 壓縮到成本很低了，我擁有了更多時間去探索更難的問題，我自己就透過 AI 實現前端部署 docker container，Gitlab CI/CD，跟設計師協作完成前端的 Design System，以往這些事情的學習成本都很高，AI 也幫我省下了不少時間，讓我邊做邊學。

我覺得身為工程師的「學會」不再是知道程式碼怎麼寫，或是知道要找哪個關鍵字的資料，而是在我知道「自己不知道」的情況下，我可以如何問對問題來突破難關，而我知道「自己知道」的事情，則是好幾倍的速度在完成實作。
