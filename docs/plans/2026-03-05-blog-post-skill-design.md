# Blog Post Skill 設計文件

**日期**：2026-03-05
**問題背景**：給筆記後 AI 生成的文章不像作者本人寫的，原因是 AI 只能模仿語氣，無法重現思考結構。
**解法方向**：把 AI 從「寫手」改成「訪談者」，加上基於 56 篇文章建立的 author profile。

---

## 檔案結構

```
.claude/skills/blog-post/
├── SKILL.md            # skill 邏輯與流程
└── author-profile.md   # 作者寫作風格 profile
```

---

## SKILL.md 流程

### Step 1 — 偵測輸入模式
- **對話模式**：使用者直接在對話中貼上筆記
- **檔案模式**：使用者提供路徑（e.g. `drafts/xxx.md`），skill 讀取該檔案

### Step 2 — 分析筆記
從筆記中抽出：
- 主題
- 觀察（Observation）
- 觀點/洞察（Insight）
- 例子（Example）
- 結論（Conclusion）

標記「還不夠清楚」或「缺少」的地方，準備追問。

### Step 3 — 一問一答追問
- 針對標記的缺口，一次問一個問題
- 根據使用者回答更新素材
- 追問直到內容足夠撐起一篇文章
- **原則**：AI 不自創觀點，只整理使用者說的內容

### Step 4 — 生成文章
- 參考 `author-profile.md` 的風格
- AI 負責整理、擴寫、重排，不產生新觀點
- 判斷文章類型（心得文 or 技術筆記）並套用對應結構
- 套用 Hexo frontmatter：title / date / tags（array）/ categories（string）
- `<!-- more -->` 放在 opening 後、第一個 `##` 前

### Step 5 — 存檔
- 直接寫入 `source/_posts/<slug>.md`
- slug 用英文或拼音，kebab-case

## author-profile.md 內容規格

### 句子與段落節奏
- 短句為主（10–20 字），長短混搭製造節奏
- 單句獨立成段用於強調
- 段落 2–4 句，不堆長段

### 兩種文章類型與對應結構

**心得文**
- 結構：觀察事實 → 說明原因/轉變 → 帶出觀點
- 不下「所以結論是...」，讓讀者自己得出
- Opening：場景或時間定位（「從...第一次...」「最近...」），不超過 3 句

**技術筆記**
- 結構：先講結論/定義 → 代碼示例 → 參考資料
- Opening：直接定義（「XXX 是...」）或問題陳述（「最近在做...發現...」）
- 結尾：參考連結，有時加小測驗

### 語氣與情緒
- 直接陳述事實，不戲劇化
- 情緒一句帶過，不渲染（e.g. 「很興奮地跳下去實驗了」）
- Technical terms 留英文
- 偶爾括號補充或自嘲，但不過分

### Closing 模式
- 心得文：感悟或展望，語氣平靜
- 技術文：參考連結或實用摘要
- 不寫「總結來說」「希望這篇對你有幫助」

### 常見句型
- `我認為/我發現` — 帶出個人觀點
- `但/只是/所以` — 主要連接詞
- `如果...就.../的話` — 條件邏輯
- `因為...所以.../原因是...` — 因果解釋
- `=>` — 試錯過程紀錄（技術文）
- 粗體/code 格式標記關鍵詞

### 黑名單句型（絕對不用）
- 「這不是...是...」
- 「換句話說...」
- 「總結來說...」
- 「這讓我想到...」
- 「先定義標準」→ 應保留原話，不升級詞彙

### 詞彙原則
- 用使用者的原話，不自創或升級詞彙
- 素材不足時先追問，不腦補填空

## 設計決策紀錄

| 決策 | 原因 |
|------|------|
| 使用兩個檔案（SKILL.md + author-profile.md） | author profile 需要獨立更新，不應與 skill 邏輯耦合 |
| 一問一答追問（不一次列出所有問題） | 根據回答動態調整下一個問題，效果更好 |
| 直接存到 `source/_posts/` | 少一個確認步驟，信任 AI 判斷 |
| 直接生成完整文章（不先給大綱） | 生成後再調整比看大綱更有效率 |
| Author profile 基於 56 篇文章分析 | 報告建議 20–50 篇，需要足夠樣本才能準確 |
